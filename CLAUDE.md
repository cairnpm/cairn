# CLAUDE.md

Operating guide for Claude Code (and any AI agent) working in this repo. Read fully before acting.
Human contributors: see [CONTRIBUTING.md](./CONTRIBUTING.md) and [README.md](./README.md).

**Cairn** — a self-hosted Shape Up "Product OS": a PM agent that turns a scattered stream of feedback
into a roadmap you can reason about. Nuxt 4 (`srcDir=app/`) + Nitro server, `node:sqlite`, shadcn-vue,
FSL-licensed. UI copy is French; code/comments/commits are English.

## Think before coding

- State your assumptions before implementing; surface tradeoffs; if genuinely unsure, ask.
- Prefer the **simplest** change that solves the problem — nothing speculative.
- If a simpler approach exists, say so and push back.
- Read the surrounding code first and match its patterns; don't invent new ones.

## Clean & DRY (this codebase is deliberately so)

- **Mutualise.** Before writing a component / composable / helper, check if one already exists. If you
  write the same thing twice, extract it — see the shared `app/components/*`, `app/composables/*`,
  `server/db/*`, `server/utils/*`.
- **Match the surrounding style exactly** — naming, structure, comment density.
- **Comments explain *why*, not *what*** — terse, in the existing voice. No filler, no noise.
- Delete dead code; never leave commented-out blocks.

## Types — keep `pnpm typecheck` at **0**

- **No `as any`. No `@ts-ignore`.** Narrow with guards (`if (!x) return`, `x ?? fallback`), not casts.
- A contained, **commented** `as <SpecificType>` or a single `@ts-expect-error` is allowed **only** at a
  genuine library-type boundary (e.g. wrapping Nuxt's over-typed `useFetch`) — never to silence app logic.
- `noUncheckedIndexedAccess` is on: guard array indexing and regex-group access.

## Architecture invariants (don't break these)

- **One write path.** All writes go through the gateway (`server/gateway/`) + the Nitro API. Reads use
  `useApiData`, writes use `useApiMutation` (`mutate` + declarative cache invalidation). Attribution
  comes from the **authenticated session**, never the request body.
- **LLM behind one interface** (`server/llm/provider.ts`) — a real Anthropic impl and a deterministic
  offline stub. To add a capability, extend the interface and both implementations.
- **SQLite is single-writer** — assume ONE process; never add horizontal scaling.
- **Zero native deps** — `node:sqlite`, `node:crypto`, etc. Nothing that must compile into the image.

## Security & self-host ethos

- **Zero-egress by default** — no telemetry, no phone-home. The linked product repo is grepped locally.
- **Secrets** (Anthropic key, GitHub App private key, repo PAT) are **encrypted at rest**
  (`server/utils/secrets.ts` → `getSecret`/`setSecret`), **write-only** in the UI (never read back), and
  **never** logged or written to disk in plaintext (e.g. git tokens go inline to the fetch/clone command,
  never into `.git/config`).
- Dev-only destructive endpoints must be hard-disabled in production (`import.meta.dev` guard).

## Tests & gates

- `pnpm typecheck` (must be **0**), `pnpm build`, and `INTAKE_TEST_STUB=1 pnpm test` (deterministic) must
  stay green before you're done.
- **CI-safe tests never hit the network and never touch `.data/`.** Intake routing is tested against the
  deterministic stub; the real-LLM run is opt-in (`pnpm test:intake` with a key).
- `.data/` holds the SQLite file + the server-side repo clone — gitignored. Never scan, lint, or commit it.

## Commits

- [Conventional Commits](https://www.conventionalcommits.org), English, imperative:
  `type(scope): summary` — `feat | fix | chore | docs | refactor | ci | test`.
- Body: bullet the *what* + *why*, terse. **No co-author / "Generated by" trailers.**
- One logical unit per commit. Commit or push **only when asked**; branch off `main` first.
